# Contributing

## Running the suites

```bash
# Keeper + intake (hermetic ExUnit, seams/fakes — no chain, no network)
mix deps.get && mix test

# Contracts (Foundry; includes the inheritable invariant suite + ABI pins)
cd contracts && forge test

# Mini App flow logic (mock EIP-1193 provider, DOM-free)
node --test webapp/tools/flow.test.mjs

# Real-EVM end-to-end (needs `forge build` artifacts + a local anvil)
( cd contracts && forge build )
anvil --silent &
mix run test/e2e/anvil_permit_lane.exs

# Co-versioning check (CI-enforced)
./scripts/check-version.sh
```

CI runs all of the above plus a vector-determinism diff and an attestation
positive/negative probe (`.github/workflows/ci.yml`).

## Co-versioning rule

The package version is stamped in four places — `VERSION`, `vectors/VERSION`,
`webapp/config.json` (`"version"`), and `mix.exs` — and
`scripts/check-version.sh` fails CI if any diverge. `CONTRACT_VERSION` and the
`SpendRouter.version()` literal move together only when Solidity bytes change.

## Golden-vector regeneration rule

`vectors/permit/*.json` are the committed output of
`node webapp/tools/gen-vectors.mjs` — generated by the webapp's ACTUAL
encoding module (`webapp/lib/permit.mjs`) and signed with `cast wallet sign
--data` as an independent EIP-712 implementation. Regeneration must be
byte-identical; CI re-runs the generator and `git diff --exit-code vectors/`.
If you change anything in the permit/envelope encoding, the 3-language
cross-check (webapp generator → Foundry redemption → Elixir keeper parsing)
fails until all three agree — regenerate the vectors in the same change and
let every suite prove the new bytes.

## Security invariants — never weaken

The full statements live in `README.md` ("Security invariants"); the short
list every change is reviewed against:

- The router never holds funds: one token-moving path,
  `transferFrom(user → destination)` with an exact-delivery check.
- The credit recipient (`user`) is derived structurally (permit signer /
  `msg.sender`) — never a keeper-controlled parameter.
- Destinations are beneficiary-bound derivations with that `user` as the
  beneficiary input, and must be recoverable by the beneficiary (the
  consumer proves recoverability; `EchoSpendRouter` is test-only).
- `_routeSpend` holds a reentrancy lock across the transfer; `orderId` is
  consumed before any external call — retries and reentrant tokens can
  never double-spend.
- No owner, pause, upgrade, rescue, or generic execute — pinned by the ABI
  test.
- Permit front-run tolerance checks ALLOWANCE, not the permit deadline;
  order freshness/TTL is the keeper's server-authoritative job.
- Compromised-keeper worst case stays: user-authorized amounts at user-bound
  destinations — recoverable, never redirectable.
- Keeper side: simulation before every broadcast, envelope-sender-only order
  authority, strict byte-for-byte grant validation against pinned config,
  `initData` never logged, raw platform ids never persisted, intake
  fail-closed (loopback bind by default, 401 before any work).
